Lecture 0

Introduction: Why Software Engineering Matters

From airports to Pokémon GO — understanding why good system design is not optional.

The Real Cost of Poor Software Engineering

Software engineering isn't about writing more code — it's about writing better software. Two famous real-world failures illustrate this perfectly.

Case Study 1: The 2024 Global Windows Outage

⚠ Critical Failure

In July 2024, a routine software update by CrowdStrike caused 8.5+ million Windows systems to crash worldwide. ~7,000 flights were cancelled, ~$5.4 billion in losses hit Fortune 500 companies, and $10 billion+ in global economic impact was estimated. One faulty update stopped the world for hours.

This happened because the update was pushed without adequate safeguards: no phased rollout, no rollback tested in production, no gradual blast radius control. Systems that depend on software reliability had none of the defensive engineering patterns you will learn in this course.

Case Study 2: Pokémon GO Launch Disaster (2016)

Pokémon GO launched in July 2016. Within hours, millions flooded the servers — far beyond what Niantic's load estimates assumed. Here's what went wrong technically:

  • Login, map, and Gyms became bottlenecks — critical services not designed for internet-scale traffic
  • Throughput exceeded capacity — requests were queued or dropped
  • Weak load balancing & geo-distribution caused cascading regional failures

Niantic had to throttle logins, disable features, and slow the rollout to survive. They had correct code — but poor system design.

The Load Balancer Analogy

Definition — Load Balancer

Like a traffic cop for computers: it takes incoming requests and spreads them across multiple servers so no single server gets overloaded, keeping everything fast and smooth.

Why You Should Care About SESD

You may write correct code, but real software fails because of poor engineering decisions. SESD teaches you:

Design for Survival

How to design software that doesn't break easily under real-world conditions.

Handle Change

How to handle evolving requirements without fear — change is inevitable in all software.

Prevent Cascades

How to prevent small mistakes from becoming catastrophic system-wide failures.

What You'll Be Able to Do After This Course

  • Design clean, structured code using classes, objects, and OOP principles
  • Think beyond writing code and reason about software behaviour and boundaries
  • Model systems clearly using ER diagrams and UML from real requirements
  • Identify and fix design problems using common design patterns
  • Write tests that validate behaviour, not just output
  • Build a small but complete MVP from requirements to working software
  • Refactor code safely as requirements change
  • Understand how software evolves through real development processes (SDLC)

Careers Where SESD Is Critical

Software Engineer (Backend / Full-Stack) System Engineer Platform / Infrastructure Engineer Tech Lead Software Architect

The Software Project Lifecycle (Overview)

Systems are built in stages, not in one shot. Skipping steps today creates problems tomorrow:

Design → Code → Test → Deploy → Maintain

This course will touch every phase of this lifecycle. Think of building a house: you don't just start hammering walls. You need a blueprint, materials list, building order, and inspection.

Lecture 1

The Three Pillars of Modern Systems

Reliability, Scalability, and Maintainability — the lens through which professional software is judged.

Five Forces Reshaping Software

Modern software engineering has been fundamentally transformed by five forces that demand deeper principles, not just surface trends:

  1. Scale Has Exploded — Internet giants handle massive data volumes and traffic. Traditional tools couldn't cope; new ones were created out of necessity.
  2. Business Velocity Demands Agility — Companies must test hypotheses cheaply and respond to market insights quickly. Development cycles are shorter; data models must be flexible.
  3. Open Source Has Won — The best minds collaborate globally on shared problems. Free and open source software is now preferred over commercial solutions.
  4. Hardware Has Changed — CPU clock speeds barely increase anymore. Multi-core processors are standard; parallelism is the only path to performance.
  5. Distribution is the New Normal — Small teams can now build globally distributed systems. Geographic distribution is expected, not exceptional.
The Result

Systems must be designed with reliability, scalability, and maintainability from day one — not bolted on later.

Pillar 1: Reliability

Definition

The system continues to work correctly even in the face of adversity — including hardware faults, software faults, and human errors.

What Adversity Looks Like

Hardware Failures

Hard disks crash (MTTF: 10–50 years per disk). RAM becomes faulty. Power grids fail. At scale, these happen constantly.

Software Errors

Bugs that crash under specific input. Runaway processes. Cascading failures where one component triggers others. Slow responses causing timeouts.

Human Errors

Configuration mistakes. Deploying bad code to production. Accidental data deletion. Security vulnerabilities.

How to Achieve Reliability

  1. Design for Failure (Fault Tolerance) — Assume every component will fail. Use redundancy: multiple servers, multiple data centers. Netflix's Chaos Monkey randomly kills production servers to test this.
  2. Automated Recovery — Detect failures quickly. Automatically restart crashed processes. Failover to backup systems without human intervention.
  3. Comprehensive Testing — Unit tests, integration tests, end-to-end tests. Chaos engineering: deliberately inject failures. Load testing to find breaking points.
  4. Minimize Human Error — Make it easy to do the right thing. Provide sandbox environments. Implement fast rollback mechanisms.

Pillar 2: Scalability

Definition

As the system grows — in data volume, traffic volume, or complexity — there should be reasonable ways of dealing with that growth.

Vertical vs. Horizontal Scaling

AspectVertical Scaling (Scale Up)Horizontal Scaling (Scale Out)
What it isBuy a bigger, more powerful machineAdd more machines, distribute load
Code changesNone — simpleMore complex — distributed systems
CostExpensive; hits a ceilingCost-effective at scale
Fault toleranceSingle point of failureBetter — redundancy built in
LimitHard physical limitsScale indefinitely (theoretically)

Common Scalability Patterns

  1. Load Balancing — Distribute requests across multiple servers
  2. Caching — Store frequently accessed data in fast memory; reduces database load 90%+
  3. Database Sharding — Split data across multiple databases
  4. Asynchronous Processing — Use message queues; don't make users wait
  5. CDNs — Serve content from servers close to users

Pillar 3: Maintainability

Definition

The ability of a system to be understood, modified, tested, and extended with low cost and low risk over time.

Most system design problems fail after launch, not at launch. Systems live much longer than originally planned, and most engineering effort is spent on changes, not greenfield development.

Core Principles of Maintainable Design

  • Separation of Concerns — Business logic ≠ infrastructure. API ≠ data storage.
  • Modularity — Components change independently of each other.
  • Simplicity — Clear is better than clever. Boring solutions age better than clever ones.

Pillar 4: Durability

Definition

The ability of a system to preserve data correctly and permanently, even in the presence of crashes, power failures, network partitions, or process restarts. Once a system acknowledges a write, it should never be lost.

Durability is especially critical for financial systems, user data, and logs/events/transactions. Data loss erodes trust and can be catastrophic.

Classes & Objects

The blueprint and the building — the fundamental building blocks of Object-Oriented Programming.

The Blueprint Analogy

Imagine you want to build 100 cars in a game like BGMI. Would you code each car individually? Of course not. You'd create a blueprint and use it to manufacture each car. That blueprint is a Class.

Definition — Class

A class is a blueprint or template that defines two things: Attributes (data/variables — what the object has) and Behaviours (functions/methods — what the object does).

Definition — Object

An object is a real instance of a class. It's the actual item created from the blueprint — something that occupies memory, has its own state and behaviour, and you can interact with.

A class is only a plan — the object is the actual item created from that plan.

Example: A Car Class

Properties a car has: Model, Color, Number of Wheels, Seating Capacity. Functionalities a car can perform: Start, Stop, Accelerate, Brake.

class Car {
  model: string;
  color: string;
  numWheels: number;
  seatingCapacity: number;

  constructor(model: string, color: string, capacity: number) {
    if (capacity < 1) throw new Error("Capacity must be >= 1");
    this.model = model;
    this.color = color;
    this.numWheels = 4;
    this.seatingCapacity = capacity;
  }

  start(): void { console.log(`${this.model} started`); }
  stop(): void  { console.log(`${this.model} stopped`); }
}

Constructors

Definition — Constructor

A constructor is a special function that runs automatically when an object is created. It sets the initial state of the object and ensures the object is in a valid, usable condition.

If you don't write a constructor, the compiler writes a default one for you: constructor() {}. This means the object is created with undefined/default values — which can be dangerous.

Invariants

Definition — Invariant

An invariant is a condition that must always remain true for an object to be in a valid state. For example: "A car must always have at least 1 seat" — seatingCapacity >= 1 is an invariant.

Invariants are enforced in the constructor and any setter methods. Why they matter:

  • Consistent and predictable object behaviour
  • Fewer bugs through early error detection
  • Better maintainability with clear rules

UML Class Diagram Basics

UML (Unified Modelling Language) is a visual modelling language. Classes are depicted as boxes with three compartments:

┌─────────────────────────────┐ │ Car │ ← Class Name (bold, centered) ├─────────────────────────────┤ │ - model: string │ ← Attributes (visibility + type) │ - color: string │ │ - seatingCapacity: number │ ├─────────────────────────────┤ │ + start(): void │ ← Methods (visibility + return type) │ + stop(): void │ │ + accelerate(): void │ └─────────────────────────────┘ Visibility: + public - private # protected ~ package

Key insight

  • Only Classes define the blueprint — they don't occupy memory
  • Only Objects occupy memory — they are instances created from classes

OOP Introduction: Inheritance & Composition

How classes relate to each other — and why the Is-A vs Has-A distinction is one of the most important design decisions you'll make.

What is OOP?

Definition — Object-Oriented Programming

OOP is a programming paradigm based on the idea of objects. Each object contains data (attributes — what it has) and code (methods — what it does). OOP focuses on data + behaviour together, not just functions floating around.

Procedural vs. OOP

AspectProcedural ProgrammingObject-Oriented Programming
FocusFunctions/procedures operating on dataObjects combining data and behaviour
ApproachTop-down, step-by-stepModel real-world entities and interactions
ExamplesC, Pascal, Fortran, BASICJava, TypeScript, Python, C#
ScalabilityWorks for small apps; breaks easilyDesigned for large, growing systems
Code reuseDifficult — logic is scatteredNatural — via inheritance and composition

Why OOP Beats Procedural for Large Systems

Without OOP: one small change → ten files break → debugging feels like archaeology. With OOP: modify only the affected objects, the rest of the system stays safe.

Inheritance — The "Is-A" Relationship

Definition — Inheritance

Inheritance allows one class to inherit the properties and methods of another class. It represents an "IS-A" relationship where a subclass IS-A type of its superclass.

When to Use Inheritance (Is-A)

  • The subclass is a specialized version of the parent — e.g. Dog is a specific type of Animal
  • Shared characteristics exist — all Animals eat, sleep, move; Dog inherits these automatically
  • The relationship is permanent — a Dog will always be an Animal, it never changes
class Animal {
  name: string;
  eat(): void { console.log(`${this.name} is eating`); }
  sleep(): void { console.log(`${this.name} is sleeping`); }
}

class Dog extends Animal {         // Dog IS-A Animal
  bark(): void { console.log("Woof!"); }
}

Common Pitfalls of Inheritance ⚠

Watch Out

1. Deep hierarchies — Long chains of classes. Changing the base class can break everything. 2. Over-generalization — Base classes trying to do too much, confusing developers. Favor composition over inheritance for flexibility and safer design.

When NOT to Use Is-A

  • When it's really a "Has-A" relationship (Dog HAS-A collar, not IS-A collar)
  • When you're just reusing code without true specialization
  • When the subclass doesn't fit the parent's behavior (Penguin IS-A Bird, but can't fly)

Composition — The "Has-A" Relationship

Definition — Composition

Composition is when a class contains or uses another class as a part. A class does NOT become another class; it simply USES it. "Dog HAS-A Tail" — no need to make Tail a Dog, just let it wag itself.

Is-A vs Has-A — Side by Side

AspectInheritance (Is-A)Composition (Has-A)
Relationship"You are this thing""You have this thing"
ExampleDog extends AnimalDog has a Tail
Change impactOne wrong change → many breakChange stays in one place
CouplingTight — child depends on parentLoose — components are independent
KeywordextendsField/reference in class
Rule of thumb: When in doubt, prefer composition. It's more flexible and safer. Inheritance is powerful but leads to rigidity if the "Is-A" relationship is weak.

Interfaces — Contracts for Behaviour

Definition — Interface

An interface is a contract, not an implementation. It says what a class can do, not how it does it. Classes implement interfaces using the implements keyword.

interface Pet {
  name: string;
  play(): void;
}

class Dog implements Pet {
  name: string;
  play(): void { console.log("Fetching ball!"); }
}

Interface vs Abstract Class

AspectInterfaceAbstract Class
What it isWhat you must doWhat you are + some default behaviour
Use caseDesign problems — loose couplingCode reuse — shared implementation
UML arrowDashed line with hollow arrow (Realization)Solid line with hollow arrow (Generalization)
Rule of thumbNeed flexibility? Use InterfaceNeed shared code? Use Abstract Class

Polymorphism & Abstraction

How one thing can do many different jobs — and how to use something without knowing how it works.

What is Polymorphism?

Definition — Polymorphism

Poly = Many, Morphism = Forms. Polymorphism is the ability of objects to take multiple forms. One interface, multiple implementations. Key benefit: write flexible and reusable code.

Real-Life Example: The Coffee Machine

Think of a coffee machine with one button labeled "BREW":

  • Press BREW → Select Espresso → Get Espresso
  • Press BREW → Select Cappuccino → Get Cappuccino
  • Press BREW → Select Latte → Get Latte

Same button (interface), different outputs (behaviour) — that's polymorphism.

Two Types of Polymorphism

1. Compile-Time (Method Overloading)

Same method name, different parameters. The decision of which method to call is made during compilation.

class Printer {
  print(msg: string): void;
  print(num: number): void;
  print(num: number, count: number): void;
}
2. Runtime (Method Overriding)

Parent-child class relationship. The correct method is determined during execution based on the actual object type.

class Animal { sound(): string { return "..."; } }
class Dog extends Animal {
  sound(): string { return "Woof"; }
}
class Cat extends Animal {
  sound(): string { return "Meow"; }
}

Benefits of Polymorphism

  • Code Reusability — Write once, use many times
  • Flexibility — Easy to add new features without modifying existing code
  • Maintainability — Changes in one place reflect everywhere
  • Cleaner Code — Less redundancy, more organised

What is Abstraction?

Definition — Abstraction

Abstraction is hiding complex implementation details and showing only essential features. Focus on "WHAT it does" not "HOW it does it". Like using a TV remote — you don't need to know the electronics inside.

Abstraction vs. Encapsulation

ConceptFocusExample
AbstractionHiding complexity from the outsideYou see "Drive()" — not how the engine works
EncapsulationProtecting data inside the classPrivate fields — only setters can modify them

Achieving Abstraction in TypeScript/Java

Abstraction is achieved through Abstract Classes and Interfaces. An abstract class cannot be instantiated directly; it forces subclasses to implement abstract methods:

abstract class Shape {
  abstract area(): number;      // Must be implemented by subclasses
  
  describe(): void {            // Concrete method shared by all
    console.log(`Area = ${this.area()}`);
  }
}

class Circle extends Shape {
  constructor(private radius: number) { super(); }
  area(): number { return Math.PI * this.radius ** 2; }
}

class Rectangle extends Shape {
  constructor(private w: number, private h: number) { super(); }
  area(): number { return this.w * this.h; }
}
Lecture 5

Encapsulation, Visibility & Invariants

Protecting system invariants through access control — the guardian of object integrity.

Encapsulation

Definition — Encapsulation

Encapsulation is the bundling of data (attributes) and the methods that operate on that data into a single unit (class), and restricting direct access to some of the object's components. It is the mechanism of hiding internal implementation details.

Visibility Modifiers

ModifierUML SymbolAccess Level
public+Visible to ALL classes
private-Visible ONLY within the same class
protected#Visible to the class and its subclasses
package/default~Visible to classes in the same package

DTOs vs Domain Objects

DTO — Data Transfer Object

A simple data-only container. No business logic, no rules. Used to transfer data between layers (e.g., API request/response). Fields are usually public or have plain getters/setters.

Domain Object

An object that enforces business rules (invariants). Private fields with controlled setters. Maintains object validity at all times. e.g., a BankAccount that won't allow negative balance.

System Forces Enabled by Encapsulation

  • Scalability — Internal implementation can change without breaking callers
  • Reliability — Invariants ensure objects are always valid
  • Observability — You control what state is visible to the outside world

UML Part 1: Class Diagrams

The visual blueprint language of software — showing what the system IS.

What is UML?

Definition — UML

UML (Unified Modelling Language) is a visual modelling language for system design. Before UML (standardised in 1997), every company used different notation — UML unified these into one standard. It is technology-agnostic: works for Java, TypeScript, Python, C#, etc.

Three Core Purposes of UML

  1. Clarifies Thinking — Forces us to formalise vague ideas before coding
  2. Communicates Requirements — Pictures speak louder than words; helps non-technical stakeholders
  3. Early Discovery — Finds structural problems before a line of code is written

Types of UML Diagrams

Structural Diagrams (What the system IS)

Focus on static components and relationships. Question: "What parts make up the system?"
Example: Class Diagram

Behavioural Diagrams (How the system BEHAVES)

Focus on dynamic processes and interactions. Question: "What does the system do over time?"
Examples: Use Case Diagram, Sequence Diagram

Class Diagrams

A UML class diagram shows the structure of a system by displaying its classes, their attributes, methods, and the relationships between them.

Class Notation (3 Compartments)

Attributes (visibility name: type) │ - privateAttr: Type │ │ # protectedAttr: Type │ ├──────────────────────────┤ │ + methodName(): RetType │ ← 3. Methods (visibility name(params): return) │ - helperMethod(): void │ └──────────────────────────┘

Relationships Between Classes — All 8 Types

UML Arrow Rules to Remember

Aggregation vs. Composition

Aggregation (Hollow ◇)

"Has a" relationship where the part can exist without the whole. Example: A Department has Professors, but Professors exist even if the Department is removed.

Composition (Filled ◆)

"Has a" relationship where the part cannot exist without the whole. Example: A House has Rooms; if the House is destroyed, the Rooms cease to exist too.

UML Part 2: Sequence Diagrams

The runtime story — showing who calls whom, in what order, over time.

Class vs. Sequence Diagrams

Class Diagram

Static blueprint — tells you what exists: structure, relationships, attributes, methods. Like an architectural blueprint of a building.

Sequence Diagram

Runtime story — tells you what happens: who calls whom, in what order. Like the step-by-step construction sequence. Removes ambiguity about flow.

A class diagram does NOT tell you runtime flow, order of interactions, or time sequence. That's exactly what a sequence diagram does.

Sequence Diagram Components

1. Actor

Actor

An external user or system that starts the interaction with our system. Represented as a stick figure. Example: Customer — the person who opens the app and clicks "Place Order".

2. Lifeline

Lifeline

Represents an object or component that exists over time during the interaction. Each lifeline is one vertical dashed line. Answers: "Who is alive in this story?"

3. Activation Bar

Activation Bar

Shows the time period when an object is busy executing logic. Represented as a narrow rectangle on the lifeline. "This object is currently doing work."

4. Message (Solid Arrow →)

Message

A method call or request sent from one lifeline to another. Every solid arrow in a sequence diagram is basically one method call between objects.

5. Return (Dashed Arrow ----→)

Return

The response sent back after a method finishes execution. Solid arrow = request. Dashed arrow = result/response.

6. Alternative Fragment (alt)

alt (Conditional)

Used to show conditional paths — only one branch executes. It's literally an if-else block in diagram form. Used for: "Restaurant is open" vs "Restaurant is closed".

Example: Place Order Sequence

Customer OrderService RestaurantService Database | | | | |--placeOrder()→ | | | | |--isOpen()-----→ | | | | | | | | [alt: open] | | | | ←--open------ | | | |--save(order)------------------→ | | | ←--orderId---------------------- | | ←--orderConfirmed()--| | | | | | | | | [alt: closed] | | | | ←--closed---- | | | ←--"Restaurant Closed"--| | |

Key rule: If you can narrate a feature step-by-step ("First the user clicks X, then the system calls Y, then..."), you can convert that narrative directly into a correct UML sequence diagram.

ER Diagrams — Part 1

Entity-Relationship modelling — the language for designing databases.

What is an ER Diagram?

Definition — ER Diagram

An Entity-Relationship (ER) Diagram is a visual representation of the data model of a system. It shows the entities (real-world objects), their attributes, and the relationships between them.

Core Components

ComponentShape (Chen Notation)Meaning
EntityRectangleA real-world object with identity (e.g., Student, Course)
AttributeEllipse/OvalA property describing an entity (e.g., name, email)
RelationshipDiamondAn association between entities (e.g., "Enrolls In")
Primary KeyUnderlined attributeUniquely identifies each entity instance
Multi-valued AttributeDouble ellipseCan have multiple values (e.g., phone numbers)
Derived AttributeDashed ellipseComputed from other attributes (e.g., age from DOB)
Composite AttributeEllipse with sub-ellipsesMade of parts (e.g., Full Name = First + Last)

Types of Entities

Strong Entity

Exists independently. Has its own primary key. Represented by a single rectangle.
Example: Student (identified by student_id)

Weak Entity

Depends on another entity for its identity. Has a partial key (discriminator), not a full primary key. Represented by a double rectangle. Its identifying relationship is shown as a double diamond.
Example: Dependent (identified by name + employee_id)

Cardinality (Relationship Constraints)

Cardinality defines how many instances of one entity can be associated with instances of another:

CardinalitySymbolMeaningExample
One-to-One (1:1)1—1One instance on each sidePerson — Passport
One-to-Many (1:N)1—NOne on left, many on rightDepartment — Employee
Many-to-One (N:1)N—1Many on left, one on rightEmployee — Department
Many-to-Many (M:N)M—NMany on both sidesStudent — Course

Participation Constraints

Total Participation (||)

Every instance of the entity must participate in the relationship. Shown as a double line. Example: Every employee MUST belong to a department.

Partial Participation (|)

Some instances may NOT participate in the relationship. Shown as a single line. Example: Not every employee manages a department.

Lecture 9

ER Diagrams — Part 2: Advanced Concepts

Associative entities, primary key selection, validation, and converting ER diagrams to relational tables.

Associative Entities (Bridge Tables)

Definition — Associative Entity

An associative entity (also called a linking entity or bridge entity) represents a relationship between two entities that has its own data. It resolves a Many-to-Many (M:N) relationship.

When Do You Need an Associative Entity?

  1. When a relationship is many-to-many (M:N)
  2. When the relationship has its own attributes
  3. When each relationship instance must be uniquely identified

Example: Student Enrolls In Course

❌ Wrong Design (without associative entity)

Student—(Enrolls In)—Course as M:N with no bridge entity. Problems: Where do you store enrollment_date? Where does the Grade go? How do you uniquely identify one enrollment?

✅ Correct Design (with associative entity)

Add an Enrollment entity. Student (1) → many Enrollments. Course (1) → many Enrollments. Enrollment has its own attributes: enrollment_date, grade. The M:N becomes two 1:N relationships.

Primary Key Selection

Definition — Primary Key

An attribute (or set of attributes) that uniquely identifies each entity instance. Must be: Unique, Non-null, and Stable (does not change over time).

TypeNatural KeySurrogate Key
DefinitionComes from real-world data, has business meaningSystem-generated, no business meaning
Exampleemail, ISBN, SSNstudent_id (auto-increment), UUID
StabilityCan change over timeArtificial and stable
RecommendationUse only if guaranteed unique AND stablePreferred in most cases
Use a surrogate key unless a natural key is guaranteed to be unique and stable. For a Student entity, email is a natural key, while student_id is a surrogate key. student_id is preferred because it remains stable over time.

Validating Your ER Diagram

A good ER diagram should make common system queries natural and easy to answer. Validation process:

Requirements → ER Draft → Sample Queries → Adjust Model

ER Diagram Validation Checklist

CheckQuestion to Ask
Entity CheckHave all real-world objects been modelled as entities?
Relationship CheckAre cardinalities (1:1, 1:N, M:N) clearly defined?
Additional Entity CheckDoes any relationship have its own attributes? (→ needs associative entity)
Attribute CheckAre all required attributes present?
Query ValidationCan the ER diagram answer all common system queries?

Converting ER Diagram to Relational Tables

After designing an ER Diagram, it must be converted into relational tables for implementation in RDBMS like MySQL, Oracle, etc.

Rule #ScenarioTables Required
Rule 1Strong entity with simple attributes1 table with all attributes; PK = key attribute
Rule 2Strong entity with composite attributes1 table; use sub-attributes (not composite itself)
Rule 3Strong entity with multi-valued attributes2 tables: one main + one for multi-valued attr
Rule 4Relationship set with own attributes1 table with PKs of both entities + own attributes
Rule 5Binary M:N relationship3 tables: A, B, and a junction table R(a1, b1)
Rule 6Binary 1:N relationship2 tables: A and BR (combining B with relationship attrs)
Rule 7Binary N:1 relationship2 tables: AR and B
Rule 8Binary 1:1 relationship2 tables: Merge relationship with either A or B
Rule 9Weak entity set2 tables: parent entity + weak entity with parent's PK

Design Patterns: Introduction & Creational Patterns

Battle-tested recipes for solving recurring software design problems.

What Are Design Patterns?

Definition — Design Pattern

Design patterns are well-proven solutions to commonly occurring problems in software design. They are not final designs but templates or recipes — think of them as a recipe book for coding. You don't reinvent how to make biryani; you follow a proven recipe.

Key Characteristics

  • Not a final design — a template or recipe that can be adapted
  • Reusable solution to problems that occur repeatedly
  • Battle-tested by the software development community
  • Provide a common vocabulary between developers

The Four Forces Behind Every Pattern

Flexibility

Can we adapt to new requirements without rewriting everything?

Coupling

Can we change one module without affecting others?

Efficiency

Are there performance considerations we must account for?

Testability

Can we write effective unit tests against this design?

The Three Pattern Families

Creational Patterns

Focus on object creation mechanisms. Manage how objects are instantiated safely and flexibly.

Structural Patterns

Focus on object composition and relationships. How classes/objects combine to form larger structures.

Behavioral Patterns

Focus on communication and interaction between objects. Manage responsibilities between objects.

Singleton Pattern — "One King to Rule Them All"

Singleton
Creational Pattern
Definition

The Singleton pattern ensures that a class has exactly one instance, and provides a global access point to that instance throughout the application.

Real-World Analogy

Netflix Account — ONE account, but you can watch on any device (phone, laptop, TV). The account is shared globally; there's only one.

Why Only One?

  • Centralized Control — Resources like configuration, logging, cache, or DB connections are managed consistently
  • Prevents Conflicts — Multiple instances may lead to state mismatches
  • Efficient Memory Use — Some objects are costly to create (DB connections)

Three Key Components

  1. Private Constructor — No one can use new to create instances directly
  2. Static Field — Holds the single instance, shared across all usage
  3. Static Access Method: getInstance() — Creates it on first request (lazy initialization); returns existing instance on subsequent calls
class Logger {
  private static instance: Logger | null = null;
  private logs: string[] = [];

  private constructor() {}  // ← Private! No "new Logger()" allowed

  static getInstance(): Logger {
    if (!Logger.instance) {
      Logger.instance = new Logger();
    }
    return Logger.instance;
  }

  log(message: string): void {
    this.logs.push(message);
    console.log(`[LOG] ${message}`);
  }
}

// Usage
const logger1 = Logger.getInstance();
const logger2 = Logger.getInstance();
console.log(logger1 === logger2); // true — same instance!

Singleton's Pitfalls

⚠ Pitfalls

Global State: Acts like a global variable — anyone can access and modify it. Unit Testing Nightmare: Cannot mock or replace the Singleton in tests. Tests become interdependent — one test affects another. Many developers now consider classic Singleton an anti-pattern.

Factory Pattern — "The Object Chef"

Simple Factory
Creational Pattern
Definition

The Factory pattern centralises object creation. A single method or class creates objects based on input parameters, decoupling client code from concrete classes.

Structure

  • Abstraction/Interface — Defines what all products must support
  • Concrete Products — The actual implementations (EmailNotification, SMSNotification)
  • Simple Factory — The factory method that creates the right product

Why We Need Factory Pattern

  • Object creation logic becomes complex
  • Need to delegate creation to subclasses
  • Want to hide the actual class type from the client

Benefits

  • Change in ONE place — update config or user preference
  • Add new types easily — Telegram, Discord, Slack (factory handles it)
  • Test different implementations — can easily swap for testing
  • No code changes in 200+ places — client code stays untouched
interface Notification {
  send(message: string): void;
}
class EmailNotification implements Notification {
  send(msg: string) { console.log(`Email: ${msg}`); }
}
class SMSNotification implements Notification {
  send(msg: string) { console.log(`SMS: ${msg}`); }
}

// The Factory
class NotificationFactory {
  static create(type: 'email' | 'sms'): Notification {
    if (type === 'email') return new EmailNotification();
    if (type === 'sms')   return new SMSNotification();
    throw new Error("Unknown notification type");
  }
}

// Client code — doesn't know or care which class is used
const notifier = NotificationFactory.create('email');
notifier.send("Your order is confirmed!");

Structural Patterns: The Adapter Pattern

Making incompatible interfaces work together — without changing either side.

What Are Structural Patterns?

Structural patterns focus on how objects and classes are composed to form larger, more flexible structures. They focus on composition rather than creation.

  • Relationships between objects
  • Composition and assembly
  • Building complex systems from simple parts
  • Maintaining flexibility and efficiency

The Structural Pattern Family: Adapter, Composite, Bridge, Decorator, Facade, Proxy.

The Adapter Pattern

Adapter
Structural Pattern
Definition

The Adapter pattern creates a "wrapper" that translates one interface into another, making incompatible interfaces work together. Core Principle: Don't change existing code; wrap it instead!

The Problem It Solves

You're building an e-commerce site. All your payment code expects a PaymentProcessor interface with a pay(amount) method. Your company decides to use Razorpay, but Razorpay has makePayment(valueInDollars). The method names don't match, and you CANNOT modify the third-party library.

  • ❌ Bad option: Rewrite your entire codebase
  • ❌ Worse option: Use third-party API directly everywhere (coupling nightmare)
  • ✅ Good option: Use the Adapter Pattern

Real-Life Analogies

Travel Power Adapter

Your device has an Indian 3-pin plug. The foreign socket is US 2-pin. The power adapter converts 3-pin to 2-pin — your device works!

USB-C to HDMI Dongle

Your laptop only has USB-C ports. The projector only has HDMI. The dongle adapts USB-C to HDMI — you can present!

Implementation

// What YOUR system expects (Target Interface)
interface PaymentProcessor {
  pay(amount: number): void;
}

// Third-party library (Adaptee) — you cannot change this!
class ThirdPartyPaymentService {
  makePayment(valueInDollars: number): void {
    console.log(`Processing $${valueInDollars} via Razorpay`);
  }
}

// The Adapter — bridges the gap
class PaymentAdapter implements PaymentProcessor {
  private thirdParty: ThirdPartyPaymentService;
  
  constructor() {
    this.thirdParty = new ThirdPartyPaymentService();
  }
  
  pay(amount: number): void {  // ← Implements your interface
    this.thirdParty.makePayment(amount);  // ← Calls their API
  }
}

// Client code — unchanged!
const processor: PaymentProcessor = new PaymentAdapter();
processor.pay(500);  // Your code calls pay(), adapter calls makePayment()

Benefits

  • No Code Rewrite — Your existing code remains unchanged; only one small adapter class needed
  • Easy to Switch Vendors — Third-party API changed? Update only the adapter!
  • Testing Made Easy — Create a MockPaymentAdapter for testing without hitting real APIs
  • Open-Closed Principle — Open for extension (add new adapters); closed for modification (existing code unchanged)

Structural Patterns: The Composite Pattern

Treating individual objects and compositions uniformly — perfect for tree-structured data.

The Composite Pattern

Composite
Structural Pattern
Definition

The Composite pattern lets you compose objects into tree structures and treat individual objects (leaves) and compositions (containers) uniformly. The client doesn't need to care whether it's dealing with one item or many.

The Problem It Solves

You're building a file manager that can calculate the total size of files and folders. A folder can contain files OR other folders (nesting). Without Composite, you'd need messy if (item is file) ... else if (item is folder) ... checks everywhere.

Pattern Structure

  • Component Interface — Common interface for all objects (getSize(), delete(), etc.)
  • Leaf — Individual object with no children (File)
  • Composite — Container that holds other components (Folder)
  • Client — Code that works with the component interface — doesn't care about leaves vs composites

Real-Life Analogies

File System

Files (leaves) have a size. Folders (composites) contain files or other folders. getSize() on a folder = sum of all children recursively.

Organisation Chart

Employee (Leaf): individual contributor. Manager (Composite): has subordinates (who can also be managers). Total salary = manager's salary + all subordinates recursively.

Implementation

// Component Interface
interface FileSystemItem {
  getSize(): number;
  getName(): string;
}

// Leaf — no children
class File implements FileSystemItem {
  constructor(private name: string, private size: number) {}
  getSize(): number { return this.size; }
  getName(): string { return this.name; }
}

// Composite — holds other FileSystemItems
class Folder implements FileSystemItem {
  private children: FileSystemItem[] = [];
  constructor(private name: string) {}
  
  add(item: FileSystemItem): void {
    this.children.push(item);
  }
  getSize(): number {
    return this.children.reduce((total, child) => total + child.getSize(), 0);
  }
  getName(): string { return this.name; }
}

// Client code — treats File and Folder the SAME way
const root = new Folder("root");
root.add(new File("readme.txt", 10));
const src = new Folder("src");
src.add(new File("app.ts", 50));
src.add(new File("utils.ts", 30));
root.add(src);

console.log(root.getSize()); // 90 — no type checks needed!

Benefits

  • Uniform Treatment — No type checking (if file... else if folder...); clean, readable code
  • Easy to Add New Types — Want to add "Shortcut"? Just implement FileSystemItem — existing code works without changes
  • Recursive Operations — getSize() automatically handles any nesting depth

Behavioral Patterns: The Observer Pattern

One update → many reactions. The backbone of event-driven programming.

Structural vs. Behavioral Patterns

"Structural patterns are about the skeleton of the system. Behavioral patterns are about the nervous system — how signals, responsibilities, and logic flow between those structures."

The Observer Pattern

Observer
Behavioral Pattern
Definition

The Observer pattern defines a one-to-many dependency between objects. When one object (Subject/Publisher) changes state, all its dependents (Observers/Subscribers) are automatically notified and updated.

The Problem It Solves

Imagine you have one app screen. You add: notifications, logs, analytics, email/SMS, live UI updates. Every time something changes, you manually call all of them. Adding a new feature means touching the core code. Result: "Too many calls, messy code, hard to add new feature, one change breaks many things."

The Big Idea: One Update → Many Reactions

  • Publisher (Subject) — The source that changes (e.g., "Deadline Updated")
  • Subscribers (Observers) — Whoever wants updates (students, systems, services)

Implementation in TypeScript

// Step 1: The Interfaces (contracts)
interface Observer {
  update(data: any): void;
}

interface Subject {
  attach(observer: Observer): void;
  detach(observer: Observer): void;
  notify(): void;
}

// Step 2: Concrete Subject (the object being watched)
class TemperatureSensor implements Subject {
  private observers: Observer[] = [];
  private temperature: number = 0;

  attach(observer: Observer): void {
    this.observers.push(observer);
  }
  detach(observer: Observer): void {
    this.observers = this.observers.filter(o => o !== observer);
  }
  notify(): void {
    this.observers.forEach(o => o.update(this.temperature));
  }
  setTemperature(temp: number): void {
    this.temperature = temp;
    this.notify();  // ← Automatically notifies ALL observers
  }
}

// Step 3: Concrete Observers (they react to changes)
class AlertSystem implements Observer {
  update(temp: number): void {
    if (temp > 100) console.log(`🚨 ALERT: Temperature is ${temp}°C!`);
  }
}

class Dashboard implements Observer {
  update(temp: number): void {
    console.log(`📊 Dashboard updated: ${temp}°C`);
  }
}

// Usage
const sensor = new TemperatureSensor();
sensor.attach(new AlertSystem());
sensor.attach(new Dashboard());
sensor.setTemperature(105); // Both observers notified automatically!

Why Use Observer?

  • Loose Coupling — The Subject only knows that an object implements Observer; it doesn't care if it's a logger, email service, or database
  • Open-Closed Principle — You can add new subscriber types without changing the publisher

Common Pitfalls

⚠ Pitfalls to Avoid

Memory Leaks: If you attach an observer but never detach it, the Subject keeps a reference forever, preventing garbage collection. Update Order: Never assume Observer A will be notified before Observer B. Performance: 10,000 observers and a single notify() call could freeze your application.

Behavioral Patterns: Strategy & Template Method

Encapsulating algorithms and fixing workflows with customizable steps.

The Strategy Pattern

Strategy
Behavioral Pattern — Composition (Has-A)
Definition

The Strategy pattern defines a family of algorithms (behaviours), encapsulates each one in a separate class, and makes them interchangeable. Strategy lets the algorithm vary independently from the clients that use it.

How Google Maps Uses Strategy

Google Maps needs to calculate routes using different modes: Driving, Walking, Biking, Public Transport. Each uses a completely different algorithm. With Strategy:

  • Google Maps (Context) selects which route strategy to use
  • Google Maps does NOT know how routes are calculated
  • Route Strategies (Car, Bike, Walking) each handle their own algorithm independently
  • User can switch modes anytime — at runtime

The Code Smells Strategy Fixes

Code Smell: Massive Switch/If-Else Blocks

If payment is Credit Card → do this. Else if payment is UPI → do that. Else if PayPal... Else if Crypto... This keeps growing with every new payment method! Every new feature requires modifying the core checkout code.

Three Key Components

  1. Strategy Interface — Defines the common contract (what all strategies must do)
  2. Concrete Strategies — Individual classes implementing different ways to do the task
  3. Context — The main class that uses a strategy and can switch between them
// 1. Strategy Interface
interface PaymentStrategy {
  pay(amount: number): void;
}

// 2. Concrete Strategies
class CreditCardStrategy implements PaymentStrategy {
  pay(amount: number): void {
    console.log(`Paid ₹${amount} via Credit Card`);
  }
}
class UPIStrategy implements PaymentStrategy {
  pay(amount: number): void {
    console.log(`Paid ₹${amount} via UPI`);
  }
}
class PayPalStrategy implements PaymentStrategy {
  pay(amount: number): void {
    console.log(`Paid ₹${amount} via PayPal`);
  }
}

// 3. Context — uses a strategy
class Checkout {
  private paymentStrategy: PaymentStrategy;
  
  constructor(strategy: PaymentStrategy) {
    this.paymentStrategy = strategy;
  }
  setPaymentMethod(strategy: PaymentStrategy): void {
    this.paymentStrategy = strategy;  // ← Switch at runtime!
  }
  processOrder(amount: number): void {
    this.paymentStrategy.pay(amount);  // ← Doesn't care HOW
  }
}

// Usage
const checkout = new Checkout(new UPIStrategy());
checkout.processOrder(1500);  // Paid ₹1500 via UPI
checkout.setPaymentMethod(new CreditCardStrategy());
checkout.processOrder(2000);  // Paid ₹2000 via Credit Card

// Adding PayPal: ZERO changes to Checkout class!

The Template Method Pattern

Template Method
Behavioral Pattern — Inheritance (Is-A)
Definition

Template Method defines the skeleton of an algorithm in a base class, letting subclasses override specific steps without changing the algorithm's overall structure. Key idea: "Don't call us, we'll call you" (Hollywood Principle). The parent class controls the flow; child classes just fill in the blanks.

Real-Life Analogy: Recipe Template

  • Preheat oven to 180°C (fixed)
  • Prepare ingredients (varies: cake vs cookies)
  • Mix ingredients (varies: method differs)
  • Bake for required time (fixed)
  • Cool down (fixed)

The STEPS are fixed, but SOME ACTIONS vary depending on what you're baking.

SOLID Principles — Part 1: SRP & OCP

The five core principles of clean OOP design — your defense against code rot.

Why SOLID?

Code changes way more than it's written. Without guidance:

  • Fragile classes — one change breaks everything
  • Rigid classes — too scared to touch
  • Tangled logic — reuse becomes a nightmare (called Immobility)

SOLID gives you a roadmap and common language to survive future changes. It's guidance, not dogma.

SOLID Acronym

Single Responsibility Principle | Open-Closed Principle | Liskov Substitution Principle | Interface Segregation Principle | Dependency Inversion Principle

S — Single Responsibility Principle (SRP)

S
Single Responsibility Principle

A class should have only one reason to change. Each class should focus on one responsibility only.

Think of SRP as giving each class a single job — don't make it a jack-of-all-trades.

The Misconception

Responsibility ≠ one method. A responsibility is defined by the actor or stakeholder who would request a change. If the CFO wants to change how taxes are calculated AND the UX Designer wants to change how the invoice looks — those are two different responsibilities, even if they're in the same class.

Violation vs Fix

❌ SRP Violation✅ SRP Correct
One Student class handles: student data, report generation, and database saving Three classes: Student (data), ReportGenerator (reports), StudentRepository (DB)

Benefits of SRP

  • Easier to understand and maintain code
  • Changes in one feature do not affect others
  • Reduces bugs caused by unrelated changes

O — Open-Closed Principle (OCP)

O
Open-Closed Principle

Software entities should be open for extension, but closed for modification.

You write a core class. Where behaviour might vary, you modify the core once to accept a Strategy or Interface. In the future, you add new features by writing new code (extension), NOT by editing the old class (modification).

How to Achieve OCP

  • Interfaces
  • Abstract classes
  • Polymorphism
  • The Strategy Pattern (directly satisfies OCP)

OCP Violation vs Fix

❌ OCP Violation✅ OCP Correct
Giant switch statement for discount types. Every time you add "Senior Citizen" discount, you edit DiscountService. Use Strategy Pattern. DiscountService accepts a DiscountStrategy interface. To add "Senior Citizen" discount, just create a new class — original service untouched.
Most Design Patterns (like Strategy or Observer) exist specifically to satisfy OCP.

SOLID Principles — Part 2: LSP, ISP & DIP

Organizing relationships between classes — from safe inheritance to decoupled dependencies.
The Bridge

SRP and OCP organize the internals of classes. LSP, ISP, and DIP organize the relationships between classes.

L — Liskov Substitution Principle (LSP)

L
Liskov Substitution Principle

Subtypes must be substitutable for their base types without altering the correctness of the program.

If code works with a parent class, it must also work with ANY child class — without special checks, surprises, or broken behaviour. LSP is not about inheritance syntax; it's about behavioural promises.

Classic Violation: The Penguin Problem

class Bird {
  fly(): void { console.log("Flying..."); }
}
class Penguin extends Bird {
  fly(): void { throw new Error("Penguins can't fly!"); }  // ❌ Breaks promise!
}

Bird promises that fly() works. Penguin breaks that promise. Any code using Bird that receives a Penguin will crash.

LSP-Correct Design

interface Bird { eat(): void; }
interface FlyingBird extends Bird { fly(): void; }

class Sparrow implements FlyingBird {
  eat(): void { console.log("Eating..."); }
  fly(): void { console.log("Flying..."); }
}
class Penguin implements Bird {
  eat(): void { console.log("Eating fish..."); }
  // No fly() — no false promises!
}

Two LSP Rules to Remember

  1. Child classes must honor parent promises — When a child extends a parent, it must behave the way the parent promises
  2. Client code should not care which subtype it gets — No if (obj instanceof Penguin) checks

I — Interface Segregation Principle (ISP)

I
Interface Segregation Principle

Clients should not be forced to depend on methods they do not use.

An interface should have only related methods. A class should not be forced to implement unnecessary behaviour. Many small interfaces are better than one large ("fat") interface.

The Fat Interface Problem

// ❌ Fat Interface — forces SimplePrinter to implement scan() and fax()
interface Machine {
  print(): void;
  scan(): void;
  fax(): void;
}
class SimplePrinter implements Machine {
  print(): void { console.log("Printing..."); }
  scan(): void { throw new Error("Not supported!"); }  // ← Forced!
  fax(): void  { throw new Error("Not supported!"); }  // ← Forced!
}

ISP Fix — Split the Interface

// ✅ Segregated Interfaces
interface Printer  { print(): void; }
interface Scanner  { scan(): void;  }
interface Fax      { fax(): void;   }

class SimplePrinter implements Printer {
  print(): void { console.log("Printing..."); }  // Only what it supports!
}
class AllInOnePrinter implements Printer, Scanner, Fax {
  print(): void { console.log("Printing..."); }
  scan(): void  { console.log("Scanning..."); }
  fax(): void   { console.log("Faxing...");   }
}

D — Dependency Inversion Principle (DIP)

D
Dependency Inversion Principle

High-level modules should not depend on low-level modules. Both should depend on abstractions. Abstractions should not depend on details; details should depend on abstractions.

Depend on "what" you want to do, not "how" it is done. Business logic should not care how emails are sent, how SMS works, or which database is used.

Identifying Roles

High-Level Module

WHAT should happen. E.g., "notify a user". This is the business logic. Should not depend on low-level details.

Low-Level Module

HOW it happens. E.g., email, SMS, push notification. Implementation details that can change.

DIP Fix — Depend on Abstractions

// Abstraction
interface NotificationService {
  send(message: string): void;
}

// Low-level modules (depend on the abstraction)
class EmailService implements NotificationService {
  send(msg: string) { console.log(`Email: ${msg}`); }
}
class SMSService implements NotificationService {
  send(msg: string) { console.log(`SMS: ${msg}`); }
}

// High-level module (also depends on the abstraction — NOT on EmailService directly)
class OrderService {
  constructor(private notifier: NotificationService) {}
  
  placeOrder(item: string): void {
    console.log(`Order placed for ${item}`);
    this.notifier.send(`Your order for ${item} is confirmed!`);
  }
}

// Usage — inject the dependency
const orderService = new OrderService(new EmailService());
orderService.placeOrder("iPhone 15");
✅ Result

To switch from Email to SMS, you just inject new SMSService(). OrderService never changes. This is the power of DIP — it protects high-level logic from change.

SOLID Summary

LSPSubtypes must honor parent promisesUnsafe, unpredictable inheritance

SDLC Models: Waterfall & Agile

SDLC is the what; the process model is the how. How teams execute those steps matters enormously.

SDLC vs. Process Model

SDLC — The WHAT

The steps that must happen:
Requirements → Design → Development → Testing → Deployment → Monitoring → Maintenance

Process Model — The HOW

The way teams execute those steps. Two major approaches: Waterfall (linear) and Agile (iterative).

Waterfall Model

Definition — Waterfall

Waterfall is a sequential, linear approach where each phase must be completed before the next begins. Like a waterfall — you can only go downward, no going back.

Phase 1: Requirements

Gather ALL requirements upfront. Document everything. No going back to change this later.

Phase 2: Design

System architecture, database design, API contracts — everything specified.

Phase 3: Implementation

Write complete code based on the plan. No changes allowed.

Phase 4: Testing

Testing happens AFTER all code is written.

Phase 5: Deployment

One final release to users.

Phase 6: Maintenance

Bug fixes and small updates.

When to Use Waterfall

  • Requirements are clear and fixed from the start
  • Changes are very rarely expected during development
  • Documentation and approvals are important at every stage
  • Large, critical, or highly regulated projects

Real Examples: Banking core systems, Space programs (NASA Mars missions), Defense systems (MIL-STD-498), Large-scale enterprise infrastructure

Agile Model

Definition — Agile

Agile is an iterative, cyclic approach. Software is built in small, usable chunks called sprints (typically 1–4 weeks). Each sprint delivers a working, tested piece of functionality. Feedback is continuous.

The Agile Cycle (Per Sprint)

Plan Sprint → Design → Build → Test → Review/Feedback → Plan Next Sprint (1–4 weeks repeating cycle)

When to Use Agile

  • Requirements may change frequently during development
  • Continuous feedback and improvement needed
  • Team wants to deliver in small usable parts quickly
  • Customer involvement is high throughout the project
  • Flexibility and fast adaptation are more important than rigid planning

Real Examples: Swiggy (deploys new features multiple times per day), Netflix (thousands of changes daily with instant rollback), Instagram, Amazon (code to production in minutes)

CI/CD Pipeline

Definition — CI/CD

CI (Continuous Integration): Code is automatically tested every time a developer pushes code. Catches errors early, no manual testing. CD (Continuous Deployment): Code is automatically deployed through stages (staging → production). Reduces manual work and errors, code goes live faster, enables continuous updates.

How Code Gets to Production (Step by Step)

1. Developer writes code

Completed feature on local machine

2. Push to Git/GitHub

Version control — code is shared and versioned

3. Code Review

Senior developer checks quality, logic, and style

4. Automated Testing (CI)

Tests run automatically — unit, integration, E2E

5. Build

Code is packaged into a deployable artifact

6. Staging Deployment

Test on real servers (not production)

7. Final Testing

QA team checks on staging environment

8. Production Deployment (CD)

Goes live to users

9. Monitor

Watch for errors, performance issues, user complaints

Common CI tools: GitHub Actions, GitLab CI, Jenkins

Modern software is NEVER finished — it continuously evolves based on user feedback.

Lecture 18

Making Sure It All Works — Testing

Testing is not optional cleanup. It is a core engineering discipline that enables safe scaling.

Why Testing Exists

Software Assumptions Fail

Users behave unpredictably. Systems grow beyond developer control. Testing validates behaviour before release.

Cost of Not Testing

Bugs reach production. User trust is lost. Emergency fixes increase cost. Teams shift from building → firefighting.

Engineering Discipline

Not optional cleanup. Not only QA's responsibility. Core part of system reliability. Enables safe scaling.

"What's worse: a bug found by a developer, or a bug found by a user?"

Level 1: Unit Testing

Definition — Unit Testing

Unit testing tests the smallest testable logic in isolation. Validates expected behaviour of individual functions or methods. Fast and isolated — no external dependencies.

Why Unit Tests Matter

  • Catch logic bugs early — before they propagate
  • Enable confident refactoring — change code, run tests, know immediately if something broke
  • Prevent regression — old features don't break when new ones are added
  • Improve code quality — testable code tends to be well-structured code
"Unit tests are like saving checkpoints — when you change something, they tell you what you accidentally broke."

Test-Driven Development (TDD)

Definition — TDD

Test-Driven Development (TDD): Write the test before writing the code. Forces clear requirements, improves design clarity. The cycle: Red → Green → Refactor.

🔴 Red

Write a failing test. It fails because the code doesn't exist yet.

🟢 Green

Write the minimum code to make the test pass.

🔵 Refactor

Clean up the code while keeping all tests passing.

Unit Testing Example with Jest

// Function to test
function calculateFinalPrice(price: number, discount: number): number {
  if (discount < 0 || discount > 100) throw new Error("Invalid discount");
  return Math.max(0, price * (1 - discount / 100));
}

// Jest unit tests
describe('calculateFinalPrice', () => {
  test('applies 10% discount correctly', () => {
    expect(calculateFinalPrice(1000, 10)).toBe(900);
  });
  test('returns 0 for 100% discount', () => {
    expect(calculateFinalPrice(500, 100)).toBe(0);
  });
  test('throws on negative discount', () => {
    expect(() => calculateFinalPrice(500, -5)).toThrow("Invalid discount");
  });
});

Popular Unit Testing Frameworks

JUnit (Java) pytest (Python) Jest (JavaScript) Mocha/Chai (JavaScript)

Level 2: Integration Testing

Definition — Integration Testing

Integration testing tests the interaction between multiple components. Validates APIs, databases, and services. Focuses on communication between parts, not just individual logic. Sits above unit tests in the testing pyramid.

Why Integration Tests Fail

  • Incorrect data flow — one component sends data in the wrong format
  • API contract mismatch — the request shape doesn't match the expected shape
  • Environment differences — works on dev machine, fails on staging
  • Dependency issues — a service that's expected to be up is down

Unit vs. Integration Testing

AspectUnit TestingIntegration Testing
ScopeSingle function / methodMultiple components working together
SpeedVery fast (milliseconds)Slower (real databases, APIs)
IsolationFully isolated — uses mocksReal dependencies
CatchesLogic bugsComponent collaboration bugs
Can replace each other?No — both are required for stability

Popular Integration Testing Tools

Used for: API + database interaction, service-to-service communication, validating request–response contracts.

Postman Supertest (Node.js) REST-assured (Java)

Level 3: End-to-End (E2E) Testing

Definition — E2E Testing

End-to-End (E2E) testing tests the system from the user's point of view. Covers the entire flow: UI → Backend → Database → External Services. Focuses on user success, not code correctness.

What E2E Tests Actually Verify

  • User can complete a task from start to end
  • All services work together correctly
  • Correct data is saved and retrieved
  • User sees the expected result

What E2E Tests Catch That Others Don't

  • Broken user flows (login → add to cart → checkout sequence fails)
  • Configuration mistakes (environment variable missing)
  • Deployment issues (code deployed but assets missing)
  • Missing service connections (payment gateway not connected)

Why E2E Tests Are Slow & Costly

  • Involve real systems and environments
  • Require databases, servers, UI rendering
  • Failures are harder to trace
  • Maintenance increases with UI changes

Key Takeaways on E2E

E2E Summary

Validates real user experience. Gives highest confidence. Most expensive form of testing. Powerful when used selectively — don't test everything with E2E.

Popular E2E Testing Tools

Cypress (Web) Selenium (Web) Playwright (Web) Appium (Mobile)

Testing Tools in CI/CD Pipelines

Tests run automatically on:

  • Every code push
  • Every pull request
  • Every deployment

This prevents broken code from reaching production. CI tools that run tests automatically: GitHub Actions, GitLab CI, Jenkins.

The Testing Pyramid

/\ / \ / E2E \ ← Few, slow, expensive, high confidence /--------\ /Integration\ ← More, medium speed, tests component interaction /------------\ / Unit Tests \ ← Many, fast, cheap, tests logic in isolation /________________\

Quick Diagnostic — Which Pattern to Use?

Decision Guide

"I need only one instance" → Singleton
"I need to create objects based on input" → Factory
"Two systems have incompatible APIs" → Adapter
"I have trees of objects and need uniform treatment" → Composite
"One change should notify many objects automatically" → Observer
"I need to swap algorithms at runtime" → Strategy
"Fixed process, some steps vary" → Template Method
